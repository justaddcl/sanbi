# Product

## Register

product

## Users

Sanbi is for busy church worship leaders and volunteers who plan worship events across desktop and mobile. They may not have training in planning software, so the app should be obvious, fast, and forgiving from the first session.

The primary user is building and maintaining worship sets: creating a team workspace, creating dated sets by event type, adding sections, adding or quick-creating songs, setting keys, adding notes, attaching resources, checking usage history, searching the library, and copying formatted set text for use elsewhere.

## Product Purpose

Sanbi exists to validate the V0 worship planning loop for a single church or team user. It should make Set Detail feel like the primary planning workspace, while Dashboard acts as a home base for upcoming sets and readiness cues rather than a KPI dashboard.

Success means a worship leader can create an account and team, build a set, add sections and songs, reorder content, manage keys and notes, search by title or tag, review song details and usage history, copy a formatted setlist, and use the app effectively on mobile and desktop.

## Brand Personality

Calm, clear, lightweight, warm, useful, fast, slightly playful, and practical over decorative.

Sanbi should have Linear-like clarity, Notion-like approachable organization, and a restrained tactile personality with hints of Teenage Engineering / Native Instruments influence. It should feel prepared and human without becoming sentimental or ornate.

## Anti-references

Avoid generic SaaS dashboards, KPI cards, marketing-site layouts, purple-gradient SaaS cliches, overly rounded card walls, and bland repeated icon-card grids.

Avoid church-bulletin aesthetics, crosses, stained glass, parchment, script fonts, religious stock imagery, and any visual language that decorates the product with church cliches.

Avoid dense music-pro software surfaces unless strongly justified. Sanbi should not emphasize artist, album, BPM, time signature, recording, or performance metadata as primary V0 workflow concepts.

## Design Principles

Set Detail is the core workspace. Treat it like a planning desk and set builder, with direct editing, clear section structure, and obvious actions.

Mobile and desktop are equally important. Core workflows must work without hover, and reordering must be usable without drag-and-drop.

Prefer direct lists, rows, sections, timelines, rails, and clear actions over dashboard metrics or decorative cards.

Keep central actions visible. Do not hide add song, add section, create set, copy set text, search, or open detail actions inside menus when they are central to the current workflow.

Archive is the safe default and delete is explicit and rare. Destructive flows should explain permanence and offer archive as the safer path where relevant.

Keep V0 simple, while preserving paths for later team switching, permissions, templates, sharing, imports, and richer collaboration.

## Accessibility & Inclusion

Use WCAG AA as the baseline. Maintain practical contrast, keyboard access, visible focus, reduced-motion respect, and state that is not communicated by color alone.

All core workflows should be possible on mobile and with non-hover interactions. Interactive elements should look tappable or clickable, long labels and song titles should wrap or truncate gracefully, and empty states should be useful rather than decorative.
